# Documentation Standards

- Use Markdown as the authoritative human-readable format.
- Keep one subject per document.
- Link decisions to requirements, tasks, contracts and code.
- Update documentation in the same pull request as implementation.
- Do not silently rewrite ADR history; supersede decisions explicitly.
- Use diagrams only where they improve understanding.
- Prefer concrete examples and stable terminology.
## Required standards links

Documentation that introduces product terminology, identifiers, APIs, errors, versions or compatibility must follow and link to these standards where relevant:

- `docs/standards/PRODUCT_TERMINOLOGY.md`
- `docs/standards/NAMING_STANDARDS.md`
- `docs/standards/ERROR_STANDARD.md`
- `docs/standards/VERSIONING_POLICY.md`
